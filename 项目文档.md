# 云矜ku - 项目文档

## 第一部分：项目概览与蓝图

### 核心问题清单：

1. **一句话介绍**：这是一个基于AI人工智能的民族服饰文化数字化保护与智能传播平台，通过图像识别技术帮助用户识别民族服饰纹样，并提供沉浸式文化展示和社交分享功能。

2. **解决什么痛点**：
   - 民族服饰文化知识碎片化，缺乏系统性的数字化整理
   - 普通用户难以识别复杂的民族服饰纹样和文化背景
   - 传统文化传播方式单一，缺乏互动性和社交属性
   - 民族文化内容创作者缺乏展示和交流的平台

3. **核心目标**：
   - 实现对56个民族服饰的智能识别，识别准确率达到95%以上
   - 建立包含10,000+纹样数据的民族服饰数字档案库
   - 打造活跃的文化社区，用户日活跃度达到5,000+

4. **关键用户**：
   - 文化爱好者：探索和了解民族服饰文化
   - 民族服饰研究者：查找特定纹样和历史资料
   - 文化创作者：上传和分享自己的发现
   - 普通用户：通过拍照识别了解身边的文化元素

5. **项目状态**：已上线并持续迭代中

## 第三部分：核心功能"庖丁解牛"

### 模块1：智能服饰识别模块

1. **功能描述**：用户上传服饰照片，系统通过AI深度学习模型自动识别民族归属、纹样含义和工艺特征。

2. **操作流程**：
   - 用户点击"开始识别"或上传图片
   - 系统显示扫描动画
   - AI服务分析图片，返回识别结果
   - 展示民族名称、纹样解释、文化背景
   - 用户可分享结果或查看更多信息

3. **业务逻辑**：
   ```
   用户上传图片 → 前端验证图片格式和大小
                 → 上传到服务器 /api/recognition/analyze
                 → 后端转发到Python AI服务 :5000
                 → YOLO8模型进行特征提取和分类
                 → 返回识别结果（民族、置信度、纹样描述）
                 → 前端展示结果（动画 + 详细信息）
   ```

4. **技术实现**：
   - **核心算法**：YOLO8目标检测 + 自定义分类模型
   - **技术难点**：
     - 处理不同光照和角度的图片 → 数据增强训练
     - 识别速度优化 → 模型量化和推理优化
   - **关键接口**：
     - `POST /api/recognition/analyze` - 识别接口
     - `POST /api/recognition/health` - 健康检查

### 模块2：锦绣画廊展示模块

1. **功能描述**：以沉浸式网格布局展示56个民族的服饰图片，支持筛选、搜索和详情查看。

2. **操作流程**：
   - 用户浏览画廊首页，看到56个民族的缩略图
   - 点击任意民族，查看该民族的详细服饰展示
   - 点击"了解更多"，跳转到民族详情页
   - 支持按地区或文化特征筛选

3. **业务逻辑**：
   ```
   访问画廊 → 读取 imageConfig.ts 映射配置
            → 遍历56个民族
            → 根据配置加载对应图片
            → 图片未找到则显示默认图
            → 点击民族 → 显示详细展示页面
   ```

4. **技术实现**：
   - **核心逻辑**：民族名称到图片文件的映射
   - **技术难点**：
     - 图片懒加载优化 → Intersection Observer API
     - 响应式布局 → CSS Grid + Tailwind
   - **关键文件**：
     - `src/pages/Gallery/index.tsx` - 画廊主页面
     - `src/pages/Gallery/imageConfig.ts` - 图片映射配置

### 模块3：文化共赏分享模块

1. **功能描述**：用户可以发布图片或视频内容，添加描述、标签、位置和音乐，与其他用户互动交流。

2. **操作流程**：
   - 用户点击"发布"，选择图片或视频
   - 上传封面图（视频必需）
   - 填写描述、添加标签、选择位置
   - 选择背景音乐（可选）
   - 发布成功后，内容出现在信息流
   - 其他用户可以点赞、评论、收藏

3. **业务逻辑**：
   ```
   用户创建帖子 → 前端验证媒体格式和大小
               → 上传媒体文件到 /api/posts
               → 后端处理图片/视频，生成缩略图
               → 保存到 MongoDB (Post 集合)
               → 更新用户统计数据
               → 推送到信息流
   ```

4. **技术实现**：
   - **核心逻辑**：Multipart/form-data 文件上传 + 内容审核
   - **技术难点**：
     - 大文件上传进度显示 → XMLHttpRequest + progress事件
     - 视频封面提取 → 在浏览器端使用video标签捕获帧
     - 位置搜索集成 → 高德地图API（建议接口 + 逆地理编码）
   - **关键接口**：
     - `POST /api/posts` - 创建帖子
     - `GET /api/posts/feed` - 获取信息流
     - `POST /api/likes` - 点赞/取消点赞
     - `POST /api/comments` - 评论

### 模块4：用户认证系统

1. **功能描述**：提供完整的用户注册、登录、个人资料管理、密码找回等功能。

2. **操作流程**：
   - 用户选择"登录"或"注册"
   - 输入邮箱和密码
   - 注册需要输入验证码（邮件发送）
   - 登录成功后跳转到首页
   - 可查看/编辑个人资料、修改密码

3. **业务逻辑**：
   ```
   用户注册 → 输入邮箱和密码
            → 点击"发送验证码"
            → 系统发送邮件验证码
            → 用户输入验证码并提交
            → 后端验证 → 创建用户记录
            → 自动登录并跳转
            
   用户登录 → 输入邮箱和密码
            → 后端验证凭证
            → 生成JWT Token
            → 返回Token + 用户信息
            → 前端保存Token到Cookie
   ```

4. **技术实现**：
   - **核心算法**：JWT认证 + 密码哈希(bcrypt)
   - **技术难点**：
     - 邮件发送稳定性 → SMTP配置 + 重试机制
     - 密码安全性 → bcrypt加密 + 6位最小长度验证
   - **关键接口**：
     - `POST /api/auth/register` - 注册
     - `POST /api/auth/login` - 登录
     - `POST /api/auth/sendCode` - 发送验证码
     - `POST /api/auth/resetPassword` - 重置密码

## 第四部分：数据库"地图"

### 核心表结构：

1. **users 表（用户表）**
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `_id` | ObjectId | 用户ID |
| `username` | String | 用户名 |
| `email` | String | 邮箱（唯一） |
| `password` | String | 加密密码 |
| `avatar` | String | 头像URL |
| `bio` | String | 个人简介 |
| `isVerified` | Boolean | 是否已验证 |
| `createdAt` | Date | 创建时间 |
| `updatedAt` | Date | 更新时间 |

2. **posts 表（帖子表）**
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `_id` | ObjectId | 帖子ID |
| `authorId` | ObjectId | 作者ID（引用users） |
| `description` | String | 描述内容 |
| `mediaType` | String | 媒体类型（image/video） |
| `media` | Array | 媒体文件URL数组 |
| `cover` | String | 封面图URL（视频用） |
| `tags` | Array | 标签数组 |
| `location` | String | 位置信息 |
| `stats` | Object | 统计信息（likes, comments, collections） |
| `createdAt` | Date | 创建时间 |

3. **likes 表（点赞表）**
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `_id` | ObjectId | 点赞ID |
| `userId` | ObjectId | 用户ID |
| `postId` | ObjectId | 帖子ID |
| `createdAt` | Date | 点赞时间 |
| 唯一索引 | (userId, postId) | 防止重复点赞 |

4. **follows 表（关注表）**
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `_id` | ObjectId | 关注ID |
| `followerId` | ObjectId | 关注者ID |
| `followingId` | ObjectId | 被关注者ID |
| `createdAt` | Date | 关注时间 |
| 唯一索引 | (followerId, followingId) | 防止重复关注 |

5. **comments 表（评论表）**
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `_id` | ObjectId | 评论ID |
| `userId` | ObjectId | 评论者ID |
| `postId` | ObjectId | 帖子ID |
| `content` | String | 评论内容 |
| `parentId` | ObjectId | 父评论ID（用于回复） |
| `createdAt` | Date | 评论时间 |

**数据库关系图（简化版）**：
```
users (用户)
  ├── posts (帖子) [一对多]
  ├── likes (点赞) [一对多]
  ├── follows (关注) [一对多]
  └── comments (评论) [一对多]

posts (帖子)
  ├── likes (点赞) [一对多]
  ├── comments (评论) [一对多]
  └── collections (收藏) [一对多]

follows (关注)
  ├── followerId → users (关注者)
  └── followingId → users (被关注者)
```
